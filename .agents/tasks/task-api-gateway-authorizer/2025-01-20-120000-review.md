# API Gateway authorizer injection into RestApi OpenAPI spec

The change adds API Gateway-level authorizer configuration to the `RestApi` CDK construct. When `authentication` is provided, `patchSecuritySchemes()` injects either a Cognito User Pool authorizer or a Lambda token authorizer into the OpenAPI spec's `securityDefinitions` before the existing `patchSecurity()` distributes global security to per-operation. A new Lambda handler (`rest-api.jwt-authorizer.ts`) validates JWT signatures for the non-Cognito path, reusing the same `jsonwebtoken` + `jwk-to-pem` pattern from the existing `auth.ts` module.

**Watch for:** The OIDC discovery URL is constructed as `${issuerUrl}/.well-known` instead of the standard `${issuerUrl}/.well-known/openid-configuration`, which will fail against spec-compliant providers (confirmed). The JWKS key cache is never invalidated, so key rotation will cause persistent auth failures until the Lambda cold-starts (confirmed). The authorizer invoke permission's `sourceArn` uses `this.node.addr` as the API resource identifier, but at construction time the SpecRestApi hasn't been created yet, so this won't match the actual API's execution ARN (likely).

## High-level view

The construct patches the OpenAPI spec in two phases: `patchSecuritySchemes` defines the authorizer (Cognito or Lambda token), then the pre-existing `patchSecurity` moves the global security reference down to each operation. The JWT path creates a full Lambda function with an invoke permission and wires its ARN into the spec as a token authorizer with 300-second TTL.

The Lambda authorizer handler is largely copy-pasted from `src/lambda/auth.ts` — ~80 lines of shared logic around JWKS fetching and JWT verification are duplicated. This is structural debt, not a bug, but divergence risk is real.

The test file covers four scenarios (Cognito auth, JWT auth, no auth, per-operation security distribution) using CDK template assertions. Not tested: the JWT authorizer Lambda handler itself — token parsing, signature validation, and error paths have zero coverage.

<details>
<summary>Issues (5)</summary>

1. **Incorrect OIDC discovery URL** — `getJwksUri` constructs `${issuerUrl}/.well-known` but the OIDC standard path is `${issuerUrl}/.well-known/openid-configuration`. This will 404 against compliant providers. Fix the path.
2. **JWKS cache never invalidates** — The module-level `cacheKeys` variable persists for the Lambda container lifetime. After key rotation, all requests will fail until a cold start. Add a TTL or catch verification failures with a cache-bust retry.
3. **Authorizer sourceArn uses node.addr instead of API ID** — The permission's `resource: this.node.addr` produces a construct-tree address, not the actual API Gateway REST API ID. The permission may be overly broad or fail to match. Use `this.api.arnForExecuteApi()` or defer permission granting until after `SpecRestApi` is created.
4. **Authentication type detection via hasOwnProperty** — Using `this.props.authentication.hasOwnProperty('userpool')` to discriminate between `ICognitoAuthentication` and `IJwtAuthentication` is fragile. A JWT config that happens to have a `userpool` key would misroute. Consider a discriminated union or explicit type guard.
5. **No unit tests for JWT authorizer Lambda** — `rest-api.jwt-authorizer.ts` is a security-critical code path with zero test coverage. Token parsing, signature validation, error paths (missing Bearer prefix, expired token, unknown kid, JWKS fetch failure), and policy document generation are all untested.

## OIDC discovery path is non-standard

In `rest-api.jwt-authorizer.ts` line 67, `getJwksUri` falls back to fetching `${discoveryUri}/.well-known` when no explicit JWKS URL is provided. The OIDC Discovery specification (RFC 8414, OpenID Connect Discovery 1.0) defines the metadata endpoint as `/.well-known/openid-configuration`:

```typescript
const wellKnownUri = `${discoveryUri}/.well-known`;
```

This will return a 404 from any spec-compliant issuer (Keycloak, Auth0, Okta, Azure AD) unless the issuer happens to serve metadata at that non-standard path. The existing `auth.ts` has the same bug, but since this is new code it should be fixed here.

## JWKS key cache has no invalidation strategy

The `cacheKeys` variable at module scope caches JWKS keys indefinitely for the lifetime of the Lambda execution environment. When a provider rotates signing keys, the authorizer will fail all token validations with signature errors. Combined with the 300-second authorizer result TTL, this creates a window where new tokens signed with the rotated key are rejected until the Lambda container is recycled.

The failure mode matters more here than in `auth.ts` because this is a gateway-level control — failure is a hard 401 for all callers, not a per-request soft failure the application handler can compensate for. A retry-with-cache-bust on signature failure would mitigate this.

## Authorizer invoke permission sourceArn mismatch

The permission granted to `apigateway.amazonaws.com` uses:

```typescript
sourceArn: cdk.Stack.of(this).formatArn({
  service: 'execute-api',
  resource: this.node.addr,
  arnFormat: cdk.ArnFormat.SLASH_RESOURCE_NAME,
  resourceName: 'authorizers/*',
})
```

`this.node.addr` is the construct tree address (a hash), not the API Gateway REST API ID. The `SpecRestApi` hasn't been instantiated at this point in the constructor. The resulting ARN will be `arn:aws:execute-api:region:account:<construct-hash>/authorizers/*`, which won't match the actual API's execution ARN.

Compare with how operation Lambda permissions are granted later in the constructor — they use `this.api.arnForExecuteApi()` after the `SpecRestApi` is created. The authorizer permission should be deferred to the same location, or use a lazy reference.

## Type discrimination via hasOwnProperty

`patchSecuritySchemes` uses `this.props.authentication.hasOwnProperty('userpool')` to distinguish between `ICognitoAuthentication` and `IJwtAuthentication`. This is a pre-existing pattern from `addCustomRestResource`, but `patchSecuritySchemes` doubles down on it. A type guard function would make the discrimination explicit and testable, and prevent silent misrouting if the interfaces ever overlap.
